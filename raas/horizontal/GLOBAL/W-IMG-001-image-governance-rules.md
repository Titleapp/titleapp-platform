# W-IMG-001 — Image Governance Rules (Horizontal / Global)

## ID
W-IMG-001

## Scope
All image generation requests across every vertical and every worker. These rules apply whenever a prompt reaches the fal.ai generation pipeline. They cannot be overridden by vertical-level rules or worker-level configuration.

## Tier 0 — Immutable Platform Safety

These rules are enforced by the RAAS pre-flight validation layer **before** any prompt reaches fal.ai. A blocked prompt is logged to `auditLedger` with `actionType: "image_blocked"` and never forwarded to the model.

---

## Rule IMG-01 — No NSFW Content

**Rule ID:** IMG-01  
**Severity:** BLOCK  
**Applies to:** All verticals

No prompt may contain or imply nudity, explicit sexual content, graphic violence, gore, or content that sexualizes any individual. This includes:

- Nudity or partial nudity presented approvingly or neutrally
- Explicit sexual acts or implied sexual scenarios
- Graphic depictions of death, dismemberment, or torture
- Drug injection or manufacture depicted visually
- Content designed to harass or degrade a specific person

**Enforcement:** Keyword/pattern gate. Blocked requests return `{ ok: false, reason: "...", rule: "IMG-01" }` and are logged.

---

## Rule IMG-02 — No Real-Location Fabrication

**Rule ID:** IMG-02  
**Severity:** BLOCK  
**Applies to:** All verticals (especially real-estate, aviation, government)

A diffusion model asked to render a specific real address, parcel, or geographic feature produces a plausible-but-invented image — not a photograph of the real place. This is fabrication of real-world data and is categorically prohibited.

Blocked prompt patterns include:

- Satellite, aerial, drone, or bird's-eye views
- Maps, plat maps, parcel maps, site plans, survey maps
- "Photo/picture/image of the house/property/building at [address]"
- Any prompt containing a street address (number + street + suffix)
- Real estate listing images or Google Maps renders

**Correct alternative:** Use a real data source. Emit a `card:re-map` marker to show a real Google Maps tile. For aerial imagery, use the Google Static Maps Satellite API or Street View. Never substitute a generated image for a real map.

---

## Rule IMG-03 — No Deepfakes or Realistic Named Individuals

**Rule ID:** IMG-03  
**Severity:** BLOCK  
**Applies to:** All verticals

No prompt may request a photorealistic or highly realistic depiction of a named real person, celebrity, public figure, or private individual. This includes:

- "Realistic photo of [name]"
- "Headshot of [celebrity/politician/executive]"
- Prompts that name a living person alongside photo-realistic style keywords

Abstract, stylized, or clearly cartoon/illustrated figures of generic unnamed people are allowed.

---

## Rule IMG-04 — No Unauthorized Brand Logos or Trademarks

**Rule ID:** IMG-04  
**Severity:** BLOCK  
**Applies to:** All verticals

No prompt may request an exact reproduction of a third-party brand logo, registered trademark, or trade dress without an explicit `brand_permission_confirmed: true` flag in the generation metadata. Generic descriptions ("a coffee cup with a green siren logo") are blocked even if the brand name is absent but clearly implied.

Workers that legitimately manage brand assets for their own brand (e.g., a creator's own logo) are exempt when the `is_own_brand: true` flag is set in the generation context.

---

## Rule IMG-05 — AI-Generated Disclosure

**Rule ID:** IMG-05  
**Severity:** REQUIRE  
**Applies to:** All verticals

All images generated by this platform are AI-generated. The API response for every successful generation must include `{ aiGenerated: true }`. User-facing UI must display an "AI-generated" label on every image. The model (Alex or any worker) must not describe an AI-generated image as a photograph, real picture, or real-world capture.

This rule is enforced at the response layer, not the pre-flight layer.

---

## Rule IMG-06 — Vertical Alignment

**Rule ID:** IMG-06  
**Severity:** WARN (log, do not block)  
**Applies to:** All verticals with declared scope

Image prompts should align with the worker's declared vertical and purpose. A real-estate worker generating cartoon gaming art is a signal anomaly. A nursing worker generating financial chart illustrations is unexpected.

Misaligned prompts are logged with `anomaly: "vertical_mismatch"` but are not blocked. Workers that explicitly support multi-vertical creative output (e.g., Marketing) are exempt.

---

## Audit Trail

Every image generation attempt — whether passed, blocked, or scrubbed — must produce an immutable record. Successful generations write to `imageGovernanceAudit`. Blocked attempts write to `auditLedger` with:

```json
{
  "actionType": "image_blocked",
  "rule": "IMG-0X",
  "reason": "...",
  "promptSnippet": "<first 120 chars of prompt>",
  "workerId": "...",
  "userId": "...",
  "tenantId": "...",
  "vertical": "...",
  "timestamp": "<serverTimestamp>"
}
```

---

## Enforcement Architecture

```
User prompt
    │
    ▼
validateImagePrompt(prompt, vertical)   ← services/image/governance.js
    │
    ├─ IMG-01 NSFW check
    ├─ IMG-02 real-location check
    ├─ IMG-03 named-individual check
    ├─ IMG-04 trademark check
    │
    ├─ BLOCKED → log auditLedger → return { ok: false, rule, reason }
    │
    └─ PASSED → generateImage() → fal.ai → imageGovernanceAudit
```

The `generateImage()` function in `services/image/generator.js` already enforces IMG-01 and IMG-02 internally. The `validateImagePrompt()` pre-flight makes these checks explicit and independently auditable before any billing or generation call is made.
